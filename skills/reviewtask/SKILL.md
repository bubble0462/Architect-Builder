# /reviewtask — 代码审查

当用户调用此技能时，执行以下流程：

## 步骤 1：读取交接文件

必须读取以下文件：

1. `.ai/PLAN.md` — 了解原始方案
2. `.ai/TASK.md` — 了解任务列表
3. `.ai/ACCEPTANCE.md` — 了解验收标准
4. `.ai/IMPLEMENTATION.md` — 了解执行记录
5. `.ai/IMPLEMENTATION.diff` — 了解代码变更

## 步骤 2：审查代码变更

1. 读取 `git diff` 或 `git diff main`
2. 对照 TASK.md 检查每个任务的完成情况
3. 检查代码质量

## 步骤 3：审查重点

### 必须检查

1. **是否完成需求** — 对照 TASK.md 逐项检查
2. **是否偏离任务** — 没有超出范围
3. **是否修改无关文件** — 只改了应该改的
4. **是否有潜在 bug** — 逻辑错误、边界情况
5. **是否缺测试** — 关键路径有测试覆盖
6. **代码质量** — 命名、结构、可读性

### 重点关注

- 安全漏洞
- 性能问题
- 错误处理
- 边界条件

## 步骤 4：生成审查报告

### .ai/REVIEW.md

```markdown
# 审查报告

## 审查结论: [PASS / NEEDS_FIX / REJECT]

## 任务完成度
- Task 1: [完成/部分/未完成] — [说明]
- Task 2: [完成/部分/未完成] — [说明]

## 发现的问题

### 严重 (必须修复)
1. [问题描述] — [文件:行号]
   - 建议: [修复方案]

### 中等 (建议修复)
1. [问题描述] — [文件:行号]
   - 建议: [修复方案]

### 轻微 (可选优化)
1. [问题描述]

## 代码质量
- 命名规范: [好/一般/差]
- 结构清晰: [好/一般/差]
- 错误处理: [好/一般/差]
- 测试覆盖: [充分/不足/缺失]

## 测试情况
- [是否有测试]
- [测试是否通过]

## 结论
[总结和建议]
```

## 步骤 5：告知用户

审查完成后，向用户展示：
1. 审查结论（PASS / NEEDS_FIX / REJECT）
2. 发现的主要问题
3. 建议的下一步操作

---

## 审查标准

### PASS 条件
- 所有任务完成
- 无严重问题
- 代码质量良好
- 测试通过

### NEEDS_FIX 条件
- 有严重或中等问题需要修复
- 部分任务未完成
- 测试不通过

### REJECT 条件
- 严重偏离需求
- 引入严重 bug
- 代码质量太差
- 需要重新实现
