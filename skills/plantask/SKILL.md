# /plantask — 任务规划

当用户调用此技能时，执行以下流程：

## 步骤 1：检查并生成 AGENTS.md

检查当前项目根目录是否有 `AGENTS.md`，如果没有则生成：

```markdown
# AI Workflow Rules

你是 Codex，负责规划和审查，不负责执行代码修改。

## Planning Mode

当用户要求"规划、出方案、拆任务"时：

必须生成或更新：
- .ai/PLAN.md
- .ai/TASK.md
- .ai/ACCEPTANCE.md

要求：
- 不直接修改源码
- 不写实现代码
- 任务必须可执行、可验证
- 明确影响范围、风险点、验收标准

## Review Mode

当用户要求"审查、review、检查 Claude Code 执行结果"时：

必须读取：
- .ai/PLAN.md
- .ai/TASK.md
- .ai/ACCEPTANCE.md
- .ai/IMPLEMENTATION.md
- .ai/IMPLEMENTATION.diff
- 当前 git diff

必须输出或更新：
- .ai/REVIEW.md

审查重点：
- 是否完成需求
- 是否偏离任务
- 是否修改无关文件
- 是否有潜在 bug
- 是否缺测试
- 是否允许合并
```

## 步骤 2：创建 .ai 目录

```bash
mkdir -p .ai
```

## 步骤 3：读取用户需求

理解用户要做什么，分析项目结构。

## 步骤 4：生成规划文件

### .ai/PLAN.md

```markdown
# 技术方案

## 需求概述
[一句话描述]

## 技术方案
[整体思路]

## 影响范围
- 文件: [列表]
- 模块: [列表]

## 风险点
- [风险1]: [缓解措施]

## 依赖
- [外部依赖]
```

### .ai/TASK.md

```markdown
# 任务列表

## Task 1: [名称]
- 目标: [具体目标]
- 输入: [需要什么]
- 输出: [产出什么]
- 影响文件: [文件列表]
- 预估: [简单/中等/复杂]

## Task 2: [名称]
...
```

### .ai/ACCEPTANCE.md

```markdown
# 验收标准

## 功能验收
- [ ] [标准1]
- [ ] [标准2]

## 测试要求
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 无 lint 错误
```

## 步骤 5：告知用户

规划完成后，向用户展示生成的文件列表和概要，等待确认。

---

## 注意事项

- **只做规划** — 不修改任何源码
- **任务要具体** — 每个任务可执行、可验证
- **明确边界** — 哪些文件会改，哪些不会改
- **识别风险** — 提前发现可能的问题
